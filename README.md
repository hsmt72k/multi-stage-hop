# multi-stage-hop

Insightは短くあるべきです。 これはグラフのエッジの属性であり、クエリで返ってきてLLMのコンテキストに入るものです。「4月降水量1.3倍超→皮薄化→煮込み適性高」。これ以上の情報は不要。構造化された命題。短いほどグラフのトラバーサルが速く、LLMへのトークン数が少なく、コストが下がる。
UserContextも比較的短くてよい。 「農家A、三河、トマト、品種X、畑0.3ha、水はけやや悪、有機栽培」。構造化された属性の集合。ノードのプロパティとして格納される。
しかしUserNarrativeは圧縮してはいけない。 「いやぁ、今年は4月にようけ降ったやろ。ほんだでこのトマト、皮がぺらっぺらや」。この生の言葉を圧縮したら、先ほど議論した「二人だけの秘密」が死にます。孫が孫でなくなる。
三つの層は圧縮度が違います。
UserNarrative。 非圧縮。おじいちゃんの言葉そのまま。方言、間、表現の癖。ただしリアルタイムのクエリでは直接参照しない。孫の応答を生成するときに、直近のUserNarrativeだけを参照して口調を合わせる。
Key Event。 中程度の圧縮。「2026年4月15日、農家A、トマト皮薄、降水量平年比150%」。事実と数値。UserNarrativeから抽出された構造化データ。BigQueryに時系列で沈む。
Insight。 最大圧縮。「降水量1.3倍超＋排水性低→皮薄化」。普遍的なパターン。固有名詞も日付もない。どの農家にも適用できる知識として抽象化されている。グラフのエッジに格納。
UserContext。 構造化属性。短い。ノードのプロパティ。
つまり知識が抽象化されるほど短くなり、個人の記憶に近いほど長く残る。Insightは辞書の見出し語のように短い。UserNarrativeは日記のように長い。
この勾配があるから、リアルタイムのクエリは速く、しかしおじいちゃんの人格は失われない。グラフを辿るときは圧縮されたInsightだけが飛ぶ。孫が話すときだけ、UserNarrativeの生の言葉が表面に出てくる。
